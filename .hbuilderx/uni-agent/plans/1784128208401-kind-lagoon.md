# 设备报修与维修功能移动端重写计划

## 概述

完全重写 `pages/repair-request/` 目录下的所有页面，将 neihao-web 管理后台的完整业务逻辑迁移到 device 移动端，采用移动端卡片式 UI 风格（不用 uni-table）。

## 目标文件结构

```
pages/repair-request/
├── list.vue           # 双Tab列表入口（我的报修 + 维修记录）【重写】
├── add.vue            # 新增/编辑报修表单（mode参数区分）【重写】
├── detail.vue         # 报修详情页【新增】
├── repair-add.vue     # 新增/编辑维修记录表单（mode参数区分）【重写】
└── repair-detail.vue  # 维修记录详情页【新增】
```

共 5 个页面文件（当前 3 个 + 新增 2 个）。

## 一、list.vue — 双Tab列表入口页【重写】

### 页面描述
使用 `uni-segmented-control` 切换"我的报修 / 维修记录"双 Tab。卡片列表 + swipe-action 滑动操作 + FAB 悬浮按钮 + 上拉加载更多。

### 交互方式
- **Tab 切换**：`uni-segmented-control` 控制 `tabIndex`（0=我的报修, 1=维修记录）
- **下拉刷新**：`enablePullDownRefresh: true`
- **上拉加载**：`uni-load-more` + `unicloud-db.loadMore()`
- **左滑操作**：`uni-swipe-action` — 编辑/删除/记录维修
- **悬浮按钮**：`uni-fab`，仅在"我的报修"Tab 显示，跳转新增报修
- **点击卡片**：跳转对应详情页

### 数据来源
- Tab 0（我的报修）：`unicloud-db` 查 `medical-device-repair-request`，where `deleted==0 && creator==$uid`，orderby `request_date desc`
- Tab 1（维修记录）：先查询当前用户所有报修单 ID，再用 `unicloud-db` 查 `medical-device-repair`，where `deleted==0 && repair_request_id in (ids)`
- 联查设备名称：`@load` 事件中批量查 `medical-device` 获取 name

### 删除保护逻辑
- **删除报修单**：先查 `medical-device-repair` 是否有 `repair_request_id` 关联且 `deleted=0`，有则弹窗阻止
- **删除维修记录**：弹窗二次确认 → 软删 repari + 恢复报修单 status=1 + 设备 status=3

### 使用组件
`uni-segmented-control`, `unicloud-db`, `uni-card`, `uni-tag`, `uni-swipe-action`, `uni-fab`, `uni-load-more`, `uni-icons`, `uni-load-state`

## 二、add.vue — 新增/编辑报修表单【重写】

### 页面描述
通过 `mode` query 参数区分：`mode=add`（新增报修）和 `mode=edit&id=xxx`（编辑报修）。

### 表单字段
**新增模式（7字段）：**
| 字段 | 组件 | 必填 |
|------|------|:--:|
| 关联设备 | `uni-combox`（搜索选择） | ✅ |
| 报修日期 | `uni-datetime-picker`（默认今天） | — |
| 报修人 | `uni-easyinput` | — |
| 联系电话 | `uni-easyinput`（手机号校验） | — |
| 故障描述 | `uni-easyinput(type=textarea)` | ✅ |
| 紧急程度 | `uni-data-select`（一般/紧急/非常紧急） | — |
| 备注 | `uni-easyinput` | — |

**编辑模式额外（4字段）：**
| 字段 | 组件 |
|------|------|
| 处理状态 | `uni-data-select`（待处理/处理中/已完成/已关闭） |
| 处理人 | `uni-easyinput` |
| 处理时间 | `uni-datetime-picker` |
| 处理结果 | `uni-easyinput(type=textarea)` |

### 提交流程
- **新增**：`db.collection('medical-device-repair-request').add()` + 设备 `status→3` + 保存 `previous_status`
- **编辑**：`db.collection('medical-device-repair-request').doc(id).update()` + 若 `status→3` 则设备恢复 `status=2` + 清除 `previous_status`

### 数据交互
改用 **clientDB 直连**（`uniCloud.database()`），弃用云对象 `medical-device-repair-co`，与 neihao-web 对齐。

## 三、detail.vue — 报修详情页【新增】

### 页面描述
全字段卡片展示 + 底部固定操作栏。

### 展示内容
- **基本信息卡片**：设备名称、报修人、联系电话、报修日期
- **故障信息卡片**：故障描述、紧急程度标签、备注
- **处理信息卡片**（有处理记录时显示）：处理状态、处理人、处理时间、处理结果

### 底部操作栏
| 按钮 | 显示条件 |
|------|----------|
| 编辑 | 始终显示 |
| 记录维修 | status=1 或 2（待处理/处理中） |
| 查看维修 | 有关联维修记录时 |
| 删除 | 无关联维修记录时 |

### 删除保护
同 list.vue：先查关联维修记录，有则提示无法删除。

## 四、repair-add.vue — 新增/编辑维修记录表单【重写】

### 页面描述
通过 `mode` query 参数区分：`mode=add&request_id=xxx&device_id=xxx` 和 `mode=edit&id=xxx`。

### 表单字段（8字段）
| 字段 | 组件 | 必填 |
|------|------|:--:|
| 维修结果 | `uni-data-select`（已修复/部分修复/无法修复） | ✅ |
| 维修日期 | `uni-datetime-picker`（默认今天） | ✅ |
| 故障描述 | `uni-easyinput(type=textarea)` | ✅ |
| 故障原因 | `uni-easyinput` | — |
| 维修方式 | `uni-easyinput` | — |
| 维修人员 | `uni-easyinput` | — |
| 维修公司 | `uni-easyinput` | — |
| 维修费用 | `uni-easyinput(type=digit)` | — |
| 备注 | `uni-easyinput` | — |

### 关联信息展示
顶部只读卡片显示：设备名称、报修单故障描述（从 URL 参数接收）

### 提交流程
- **新增**：`add` 维修记录 → 若 `result=1`（已修复）则设备恢复 `previous_status` + 报修单 `status→3`；否则报修单 `status→2`
- **编辑**：`update` 维修记录 → 同新增级联逻辑

## 五、repair-detail.vue — 维修记录详情页【新增】

### 页面描述
全字段展示 + 底部操作栏（编辑/删除）。

### 展示内容
- **关联信息卡片**：设备名称、报修人、原始故障描述
- **维修信息卡片**：维修日期、维修结果标签、故障原因、维修方式、维修人员、维修公司、费用、备注

### 底部操作栏

- 删除 → 确认弹窗 → 软删 + 报修单→待处理 + 设备→维修中

## 七、pages.json 路由配置

在 `pages.json` 中配置 5 个页面路由（3 个已有 + 2 个新增）：

```json
{ "path": "pages/repair-request/list", "style": { "navigationBarTitleText": "报修与维修", "enablePullDownRefresh": true } },
{ "path": "pages/repair-request/add", "style": { "navigationBarTitleText": "" } },
{ "path": "pages/repair-request/detail", "style": { "navigationBarTitleText": "报修详情" } },
{ "path": "pages/repair-request/repair-add", "style": { "navigationBarTitleText": "" } },
```

`add.vue` 和 `repair-add.vue` 的 title 在 `onLoad` 中动态设置（根据 mode）。

## 八、页面跳转关系图

```
list.vue (双Tab入口)
  ├─ [FAB] ──→ add.vue?mode=add&device_id=xxx
  ├─ [卡片点击] ──→ detail.vue?id=xxx (报修)
  │                  ├─ [编辑] ──→ add.vue?mode=edit&id=xxx
  │                  ├─ [记录维修] ──→ repair-add.vue?mode=add&request_id=xxx
  │                  └─ [查看维修] ──→ repair-detail.vue?id=xxx
  ├─ [卡片点击] ──→ repair-detail.vue?id=xxx (维修)
  │                  └─ [编辑] ──→ repair-add.vue?mode=edit&id=xxx
  ├─ [左滑编辑] ──→ add.vue?mode=edit&id=xxx
  ├─ [左滑记录维修] ──→ repair-add.vue?mode=add&request_id=xxx
  └─ [左滑删除] ──→ 弹窗确认 → 级联软删除
```

页面间通过 URL query 参数传递数据，提交成功后 `navigateBack()` + `uni.$emit('repair-refresh')` 通知列表刷新。

## 九、业务规则汇总（参照 neihao-web）

| 操作 | 报修单变化 | 设备变化 | 维修记录变化 |
|------|-----------|----------|------------|
| 新增报修 | `status=1` | `status→3, previous_status=old` | — |
| 编辑报修为已完成 | `status→3` | `status=previous_status(remove)` | — |
| 编辑报修为已关闭 | `status→4` | `status→2` | — |
| 新增维修（已修复） | `status→3, handler/handle_date/handle_result` | `status=previous_status(remove)` | 新增 |
| 新增维修（部分/无法修复） | `status→2` | 不变 | 新增 |
| 编辑维修（已修复） | `status→3` | `status=previous_status(remove)` | 更新 |
| 编辑维修（未修复） | `status→2` | `status=3` | 更新 |
| 删除报修（无关联维修） | 软删 | `status→2` | — |
| 删除报修（有关联维修） | ❌ 阻止 | — | — |
| 删除维修 | 软删 | `status→3` | 恢复 `status=1` |

## 十、颜色/样式规范

```scss
// 主题色
$primary: #6366f1;  // 主色（按钮/导航）

// 页面
$page-bg:     #f5f5f7;  // 页面背景
$card-bg:     #ffffff;  // 卡片背景
$card-radius: 16rpx;    // 卡片圆角

// 状态映射
报修状态: 1-待处理(warning) / 2-处理中(primary) / 3-已完成(success) / 4-已关闭(info)
紧急程度: 1-一般(info) / 2-紧急(warning) / 3-非常紧急(error)
维修结果: 1-已修复(success) / 2-部分修复(warning) / 3-无法修复(error)
```
